# GUI Agent 框架与实现报告（更新于 2026-03-29）

## 1. 目标与当前迭代方法

GUI Agent 的目标是：在 Android 真实设备上，把自然语言任务拆成单步执行，并通过 Oracle 校验“动作是否带来了正确变化”，最终完成任务。

本轮执行策略（已落地到代码与 benchmark 流程）：

1. 持续跑 `main`/benchmark 真实任务（非 dry-run）。
2. 定位 Oracle 或动作层设计问题。
3. 做最小必要修复后立即复测。
4. 难例不长时间卡住：触发超时后跳过该任务，并记录失败原因。
5. 继续下一轮迭代，直到 general 任务稳定性提升。

---

## 2. 端到端框架（实际运行链路）

入口：`main.py` -> `AgentLoop.run()`。

单步循环在 `agent_loop.py::_execute_one_step()` 的实际顺序：

1. 感知当前状态：截图 + dump +（按条件）CV，构建 `UIState`。
2. Planner 产出单步 `SubGoal`（可来自长期经验回放或 LLM）。
3. Pre Oracle 生成 `PreConstraints`（变化证据计划 + 边界约束）。
4. SafetyInterceptor 执行高风险拦截。
5. ActionMapper 将 `SubGoal` 映射成可执行 `Action`。
6. Running Oracle 执行前检查（死循环检测）。
7. ActionExecutor 执行动作（ADB）。
8. 再次感知，得到 `new_state`。
9. Running Oracle 执行后检查（异常/边界/跳变偏离，分 soft/hard）。
10. Post Oracle（Evaluator）做 machine-first 评估，必要时才调 LLM 语义确认。
11. Replanner 按结果决定继续、回退、重规划或中止。

补充控制逻辑：

- `run()` 中支持任务级 `max_task_seconds`。
- Post Oracle 返回 `uncertain` 时，主循环会“等待 -> 再观测一次 -> 再评估”。
- 回退使用 `adaptive_back`：第一次 back 无状态变化会自动补第二次。

## 3. 模块与实现细节（按代码现状）

### 3.1 配置与入口

- `config.py`：集中管理 ADB/LLM/CV/memory/runtime 参数。
- 新增并生效：`max_task_seconds`（默认 0，不限时）。
- `main.py`：新增 CLI 参数 `--max-task-seconds`，透传到 `AgentConfig`。

### 3.2 Perception（感知层）

核心文件：

- `Perception/perception_manager.py`
- `Perception/dump_parser.py`
- `Perception/context_builder.py`
- `Perception/uied/detect.py`

实现要点：

- 优先解析 dump；dump 语义足够时可跳过 CV，降低时延。
- CV 与 dump 通过 IoU 融合，输出统一 `UIState`。
- `UIState` 带有：widgets、activity/package、keyboard_visible、screen_size。
- 每轮感知结果会落盘到 `data/context/*.json` 供复盘。

### 3.3 Planning（单步规划）

核心文件：`Planning/planner.py`。

实现要点：

- 单步规划，不一次性生成全链路。
- 先尝试长期经验检索（文本相似度），命中后按 progress 回放。
- 未命中时调用 LLM 生成 `SubGoal`。
- `SubGoal` 关键字段：
  - `description`
  - `action_type`
  - `target_widget_id` / `target_widget_text`
  - `input_text`
  - `acceptance_criteria`
  - `expected_transition`
  - `from_experience`

### 3.4 Pre Oracle（事前约束）

核心文件：`Planning/oracle_pre.py`。

当前 `PreConstraints` 契约：

- `action_anchor`
- `success_evidence_plan`
- `boundary_constraints`
- `semantic_goal`

实现要点：

- 快路径覆盖 `tap/input/swipe/scroll/scroll_up/scroll_down/back/enter/long_press`，减少不必要 LLM 调用。
- 新边界字段：
  - `package_mismatch_severity`（`soft|hard`）
  - `related_package_tokens`
- 支持“打开 app”语义推断目标包名（`expected_package`）。
- prompt 模板缺字段时不崩溃，回退快速约束。

### 3.5 Safety Interceptor（安全拦截）

核心文件：`Planning/safety_interceptor.py`。

实现要点：

- 按高风险关键词（支付/删除/授权等）触发人工确认。
- benchmark 中已通过 `config.high_risk_keywords=[]` 关闭交互阻塞。

### 3.6 ActionMapper（动作映射）

核心文件：`Execution/action_mapper.py`。

实现要点：

- 映射优先级：
  1. 简单动作直映射（如 back）
  2. 确定性规则映射
  3. `widget_id` 直达
  4. 文本匹配
  5. LLM 映射兜底
- 新增确定性映射：
  - Clock 底部 tab
  - Timer 键盘数字/00/backspace（含 clear/reset -> long_press）
  - Launcher 顶部搜索栏
  - Launcher 打开 App（找不到图标文本时回退 `launch_app(package)`）
- 默认滑动轨迹已增强，避免无目标滑动退化为误点击。

### 3.7 ActionExecutor（ADB 执行）

核心文件：`Execution/action_executor.py`。

实现要点：

- 支持动作：`tap/swipe/input/back/enter/long_press/launch_app`。
- `launch_app(package)`：
  - 先 `monkey -p <pkg> -c LAUNCHER 1`
  - 失败再 `resolve-activity + am start -n`。
- `dump_ui()` 已做陈旧文件防护：
  - dump 前删本地与远端旧文件。
  - 识别 `could not get idle state` 时直接返回空 dump。
  - pull 后校验文件存在与最小大小，避免误用脏 XML。

### 3.8 Running Oracle（执行期 Oracle）

核心文件：`Execution/oracle_runtime.py`。

实现要点：

- 执行前：死循环检测（连续同动作阈值）。
- 执行后：输出分级结果：
  - `severity = none|soft|hard`
  - `action_needed = none|back|replan`
- 包边界不再“一刀切”：
  - 包关系分 `same|related|unrelated`
  - `related` 漂移可 soft 放行
  - 仅在严格场景（`package_mismatch_severity=hard` + unrelated）判硬失败。

### 3.9 Post Oracle / Evaluator（事后评估）

核心文件：`Evaluation/evaluator.py`。

实现要点：

- 先机器评估，再在 `uncertain` 时调用 LLM。
- 评估输出：`decision = success|fail|uncertain` + `confidence` + `suggested_next_action`。
- 机器层流程：
  1. 边界硬约束检查
  2. 提取 `delta_facts`
  3. 对照 `success_evidence_plan` 匹配 required/supporting/counter 信号
  4. 生成 machine 决策
- 已去掉“焦点变化直接 success”捷径。
- 对弱视觉变化增加动作归因约束，降低误判成功。

### 3.10 Replanner（失败恢复）

核心文件：`Evaluation/replanner.py`。

实现要点：

- 连续失败达阈值触发 back（死胡同保护）。
- 否则调用 LLM 生成 `retry/back/replan/abort`。
- 任务成功后沉淀语义步骤到长期记忆；任务失败不沉淀。

### 3.11 Memory（短期 + 长期）

核心文件：

- `Memory/memory_manager.py`
- `Memory/experience_store.py`

实现要点：

- 短期记忆：当前任务历史、动作日志、失败原因。
- 长期记忆：成功经验 JSON 持久化，按文本相似度检索回放。

## 4. 当前关键数据契约

### 4.1 `SubGoal`

表达“这一小步要做什么”，含动作类型、目标锚点、输入、验收语义与跳变预期。

### 4.2 `PreConstraints`

把子目标转换成“可验证变化计划”：

- 锚点（`action_anchor`）
- 成功信号计划（`success_evidence_plan`）
- 边界约束（`boundary_constraints`）
- 语义目标（`semantic_goal`）

### 4.3 `Action`

执行层动作实体：坐标/滑动终点/输入文本/目标控件属性/动作类型。

### 4.4 `EvalResult`

评估输出：

- `success`（布尔）
- `decision`（`success|fail|uncertain`）
- `confidence`
- `suggested_next_action`
- `matched/unmatched/counter signals`
- `delta_facts` 与边界证据

## 5. 本轮核心变更总览（已在代码）

1. 打通任务级超时：`config/main/agent_loop` + benchmark 统计 `timeout_skipped_count`。
2. 强化 dump 稳定性：避免陈旧 dump 污染。

4. 增加多组确定性映射（Clock/Timer/Launcher），降低 CV-only 误点。
5. Pre Oracle 边界策略改造：`expected_package + package_mismatch_severity + related_package_tokens`。
6. Running Oracle 分级处理 soft/hard；不再把所有包漂移都当硬失败。
7. Post Oracle machine-first + uncertain 二次观测路径。
8. 失败原因在 benchmark 中可追踪（`failure_summary`）。

## 7. 当前仍存在的设计问题

1. 任务超时不是硬中断。
- 当前只在主循环步边界检查超时。
- 单步内部的长耗时调用（LLM/CV/OCR）会让任务超时被严重延后触发。

2. `scroll` 语义在 mapper 端仍不完全统一。
- Pre Oracle 快路径已支持 `scroll`。
- ActionMapper 的强制默认滑动分支主要覆盖 `swipe/scroll_up/scroll_down`，`scroll` 仍可能走到非预期路径。

3. Launcher 场景仍有误触与循环风险。
- 搜索栏和应用图标在噪声状态下仍可能被误映射。
- 会触发“回主屏 -> 再搜索”的循环，最终超时。

4. Timer 难例鲁棒性不足。
- 初始脏状态、OCR 读值噪声、动作归因链不稳，导致推进效率低并超时。

## 8. 现行执行策略（与你要求对齐）

1. 继续“测试 -> 发现问题 -> 最小修复 -> 复测”的闭环。
2. 对高成本难例启用任务超时，避免卡死。
3. 超时后直接跳过并沉淀失败原因，不阻塞整体迭代。
4. 以 general benchmark 成功率和 timeout 数作为主指标，逐轮收敛。

## 9. 当前状态结论

- 框架层面：`pre/running/post oracle` 三段契约已经完整落地，主循环实现闭环可运行。
- 工程层面：动作与边界策略已有明显增强，失败可观测性显著提升。
- 稳定性层面：仍处于“有阶段性提升，但尚未稳定达标”的状态，主要瓶颈在超时硬控制与 launcher/timer 难例。
