## 1. 当前框架模块（按真实执行链路）

入口：`main.py -> agent_loop.py:AgentLoop`

单步主链路（`AgentLoop._execute_one_step`）：

1. `Perception`：采集并解析当前 UI，输出 `UIState`。
2. `Planning/Planner`：生成本步意图 `PlanResult`。
3. `Planning/OraclePre`：把 `PlanResult` 编译为可执行前契约 `StepContract`。
4. `Planning/SafetyInterceptor`：高风险关键词人工确认。
5. `Execution/ActionMapper`：把计划意图落成设备可执行动作 `ResolvedAction`。
6. `Execution/OracleRuntime.pre_guard`：执行前守卫，输出 `GuardResult(pre)`。
7. `Execution/ActionExecutor`：执行 ADB / uiautomator2 动作。
8. `Perception` 再感知：输出执行后 `UIState`。
9. `Execution/OracleRuntime.post_guard`：执行后守卫，输出 `GuardResult(post)`。
10. `Evaluation/Evaluator`：融合 observations + policy + expectations，输出 `StepEvaluation`。
11. `Evaluation/Replanner`：根据 `StepEvaluation` 输出 `ReplanDecision`。
12. `Memory`：写短期轨迹（长期记忆链路当前停用）。
13. `utils/AuditRecorder`：分阶段写入 `data/*` 审计产物。

## 2. 模块数据结构与字段含义

## 2.1 Perception 模块

### `UIElement`（`Perception/dump_parser.py`）

含义：从 Android XML dump 解析出的原始控件对象（语义源）。

字段：

1. `resource_id`：控件资源 ID。  
2. `class_name`：控件类名。  
3. `text`：文本。  
4. `content_desc`：无障碍描述。  
5. `bounds`：控件边界 `(x1, y1, x2, y2)`。  
6. `clickable/scrollable/enabled/focusable/editable/focused/checkable/checked/selected`：交互与状态位。  
7. `package`：节点所属包名。  
8. `index`：节点索引。  

派生属性：

1. `width/height/area/center`：几何派生值。  
2. `is_interactive`：是否可交互。  
3. `is_editable`：是否输入类控件。  

### `WidgetInfo`（`Perception/context_builder.py`）

含义：CV + dump 融合后的统一控件对象（运行时统一控件粒度）。

字段：

1. `widget_id`：本次 `UIState` 内控件编号。  
2. `bounds`：边界框。  
3. `center`：中心点。  
4. `category`：CV 类别。  
5. `text`：展示文本。  
6. `resource_id`：资源 ID。  
7. `class_name`：类名。  
8. `content_desc`：无障碍描述。  
9. `clickable/scrollable/enabled/focusable/focused/editable/checkable/checked/selected`：控件状态。  
10. `cv_confidence`：CV 置信度。  

### `UIState`（`Perception/context_builder.py`）

含义：单步执行前后的统一界面快照，规划/执行/评估共享。

字段：

1. `widgets`：供 LLM 使用的精简视图。  
2. `full_widgets`：执行与评估使用的完整视图。  
3. `activity_name`：当前前台 Activity。  
4. `package_name`：当前包名。  
5. `screen_width/screen_height`：屏幕尺寸。  
6. `keyboard_visible`：软键盘是否可见。  
7. `screenshot_path`：对应截图路径。  
8. `raw_cv_elements`：原始 CV 结果。

### `PlanResult`（`Planning/planner.py`）

1. `goal: GoalSpec`：本步目标定义。

3. `requested_action_type`：请求动作类型。  
4. `input_text`：输入类动作文本。  
5. `planning_hints`：扩展提示（包名、方向、回退信息等）。  
6. `is_task_complete`：是否任务已完成。  
7. `reasoning`：规划理由。  
8. `from_experience`：是否来自经验回放。

## 2.3 Oracle 契约层（`Oracle/contracts.py`）

这层是全链路结构对齐中心。核心 dataclass 如下。

### `GoalSpec`

1. `summary`：本步目标一句话摘要。  
2. `success_definition`：成功标准。  
3. `tags`：标签集合。  

### `Selector`

1. `kind`：选择器类型（如 `widget_id/text/resource_id/bounds/point`）。  
2. `operator`：匹配操作（如 `equals/contains/regex/overlap/near`）。  
3. `value`：匹配值。  

### `TargetResolved`

1. `widget_id`：命中的运行时控件 ID。  
2. `bounds`：命中控件边界。  
3. `center`：命中控件中心点。  
4. `snapshot`：命中控件属性快照。  

### `TargetRef`

1. `ref_id`：目标引用 ID（例如 `target:primary`）。  
2. `role`：`primary/secondary/fallback`。  
3. `selectors`：选择器列表。  
4. `resolved`：解析后的目标结果。  

### `Predicate`

1. `field`：字段路径（例如 `attributes.similarity`）。  
2. `operator`：断言操作符。  
3. `value`：断言值。  

### `Expectation`

1. `id`：期望 ID。  
2. `fact_type`：要匹配的事实类型。  
3. `scope`：`anchor/local/global`。  
4. `subject_ref`：绑定主体引用。  
5. `predicates`：断言条件。  
6. `weight`：权重。  
7. `optional`：是否可选。  
8. `polarity`：`positive/negative`。  
9. `tier`：`required/supporting`。  

### `PolicyRule`

1. `id`：策略 ID。  
2. `kind`：策略类型（`app_boundary/activity_boundary/loop_guard/visual_guard/...`）。  
3. `level`：`none/soft/hard`。  
4. `subject_ref`：策略绑定主体。  
5. `predicates`：通用断言。  
6. `tags`：策略标签。  
7. `boundary_mode`：`stay/switch/either`。  
8. `expected_packages`：期望包名集合。  
9. `forbidden_packages`：禁止包名集合。  
10. `expected_activity_contains`：期望 Activity 片段。  
11. `loop_threshold`：循环阈值。  
12. `max_similarity`：视觉相似度阈值。  

### `StepContract`

1. `goal`：目标定义。  
2. `target`：目标引用。  
3. `expectations`：执行后期望证据。  
4. `policies`：运行时策略约束。  
5. `planning_hints`：上下文提示。

## 2.4 Execution 模块

### `ResolvedAction`（由 `ActionMapper` 生成）

含义：执行器可直接执行的标准动作对象。

字段：

1. `type`：动作类型（`tap/input/swipe/back/enter/long_press/launch_app`）。  
2. `params`：动作参数。  
3. `target`：动作对应目标。  
4. `description`：可读描述。  

`params` 常见字段约定：

1. `tap`：`x,y`。  
2. `input`：`x,y,text`。  
3. `swipe`：`x,y,x2,y2,duration_ms,screen_width,screen_height`。  
4. `long_press`：`x,y,duration_ms`。  
5. `launch_app`：`package,activity?`。  

### `MappingFailure`（`Execution/action_mapper.py`）

字段：

1. `reason_code`：失败原因码。  
2. `message`：失败文本。  

### `ObservationFact`（`Execution/observation_extractor.py` + runtime guard）

含义：统一事实证据单元。

字段：

1. `fact_id`：事实 ID。  
2. `type`：事实类型。  
3. `scope`：`anchor/local/global`。  
4. `subject_ref`：主体引用。  
5. `attributes`：事实属性。  
6. `confidence`：置信度。  
7. `source`：来源（`ui_tree/screenshot_diff/runtime_guard/action_mapper/...`）。  
8. `is_derived`：是否派生事实。  
9. `evidence_refs`：关联证据引用。  

### `Assessment`（`Execution/policy_engine.py` / `Evaluator`）

含义：统一判定条目，策略引擎和评估器都输出这个结构。

字段：

1. `name`：判定名。  
2. `source`：`runtime/policy_engine/evaluator/semantic_llm`。  
3. `applies_to`：`boundary/effect/semantic/safety/runtime_guard`。  
4. `outcome`：`pass/fail/uncertain`。  
5. `severity`：`none/soft/hard`。  
6. `reason_code`：原因码。  
7. `message`：说明文本。  
8. `evidence_refs`：证据 ID 列表。  
9. `score`：可选数值。  
10. `remedy_hint: AdviceParams`：建议动作参数。  

### `GuardResult`（`Execution/oracle_runtime.py`）

字段：

1. `phase`：`pre/post`。  
2. `allowed`：是否允许继续。  
3. `assessments`：守卫判定。  
4. `observations`：守卫阶段事实。

## 3. 整链路数据结构如何对齐

## 3.1 对齐机制

1. 单一契约源：核心结构和枚举统一在 `Oracle/contracts.py`。  
2. 结构化解析：`parse_json_object + parse_dataclass` 先解析再入链路。  
3. 动作类型收敛：`validate_action_type` 统一动作枚举。  
4. 主体引用归一：`normalize_subject_ref` 统一 `subject_ref/ref_id`。  
5. 谓词字段白名单：`FACT_ATTRIBUTE_SCHEMA` 限制 `Expectation.predicates.field` 合法路径。  
6. 统一序列化：`to_plain_dict` 确保 dataclass、list、dict 跨模块落盘一致。  
7. 审计写盘统一：`AuditRecorder.record_step` 统一 `data/<Artifact>/step_n.json` 格式。  

## 3.2 对齐关系（结构映射）

1. `PlanResult.requested_action_type` -> `ActionMapper` -> `ResolvedAction.type`。  
2. `StepContract.expectations[*].fact_type` -> `ExpectationMatcher` 按 `ObservationFact.type` 匹配。  
3. `StepContract.policies[*]` -> `PolicyEngine` -> `Assessment` -> `Evaluator` 合并。  
4. `TargetRef.ref_id/subject_ref` 贯穿 `StepContract -> ResolvedAction -> ObservationFact -> Assessment`。  
5. `RecommendedAction.kind` -> `Replanner` -> `ReplanDecision.action`。  

## 3.3 OraclePre 对齐补偿（防 LLM 漂移）

1. 若 LLM contract 缺字段，使用 fallback expectations/policies 补全。  
2. 强制存在关键 policy kind：`loop_guard/app_boundary/activity_boundary/visual_guard`。  
3. `visual_similarity_state` 不允许作为 required 硬证据。  
4. 自动补 `safety.no_risk.required`（负向期望）。  
5. `app_boundary` 会根据动作与当前包自动推导 `boundary_mode` 与 `expected_packages`。

## 4. 整链路数据结构如何生成

| 阶段 | 生成模块 | 输入 | 输出 | 主要消费方 |
|---|---|---|---|---|
| 感知前态 | `PerceptionManager.perceive` | screenshot + dump + activity/package/keyboard | `UIState(before)` | Planner / OraclePre / ActionMapper / Runtime |
| 规划 | `Planner.plan` | task + `UIState` + short memory summary | `PlanResult` | OraclePre / Safety / ActionMapper |
| 合同生成 | `OraclePre.generate_contract` | `PlanResult` + `UIState` + task_hint | `StepContract` | Safety / ActionMapper / Runtime / Evaluator |
| 安全拦截 | `SafetyInterceptor.check` | `PlanResult` + `StepContract` | bool | AgentLoop |
| 动作映射 | `ActionMapper.map_action` | `PlanResult` + `StepContract` + `UIState` | `ResolvedAction` or `MappingFailure` | Runtime / Executor |
| 执行前守卫 | `OracleRuntime.pre_guard` | `ResolvedAction` + `StepContract` + `UIState(before)` | `GuardResult(pre)` | AgentLoop |
| 设备执行 | `ActionExecutor` | `ResolvedAction.params` | 设备状态变化 | Perception |
| 感知后态 | `PerceptionManager.perceive` | after screenshot/dump + runtime state | `UIState(after)` | Runtime / Evaluator |
| 执行后守卫 | `OracleRuntime.post_guard` | action + contract + `UIState(before/after)` | `GuardResult(post)` | Evaluator |
| 后评估 | `Evaluator.evaluate` | contract + guard outputs + states + action | `StepEvaluation` | Replanner / AgentLoop |
| 重规划决策 | `Replanner.handle_failure` | `StepEvaluation` + `UIState` | `ReplanDecision` | AgentLoop |
| 记忆写入 | `ShortTermMemory.add_step/add_action` | 过程结构化产物 | history/action_log/failure | Planner（下一步上下文） |

## 5. 落盘结构与内存结构的对应关系

`AuditRecorder` 会把每一步产物写到 `data/<ArtifactKind>/step_<n>.json`。  
关键点是：有些文件直接写对象，有些用 `append=True` 写成“阶段数组”。

## 5.1 直接对象（单次覆盖写）

1. `PlanResult/step_n.json`：`PlanResult` 对象。  
2. `StepContract/step_n.json`：`StepContract` 对象。  
3. `ResolvedAction/step_n.json`：`ResolvedAction` 对象。  
4. `action/step_n.json`：同 `ResolvedAction`（兼容旧命名）。  

## 5.2 阶段数组（append）

1. `UIState/step_n.json`：元素形如 `{"phase": "...", "state": UIState}`。  
2. `GuardResult/step_n.json`：元素形如 `{"phase": "...", "value": GuardResult}`。  
3. `ObservationFact/step_n.json`：元素形如 `{"phase": "...", "facts": [ObservationFact...]}`。  
4. `Assessment/step_n.json`：元素形如 `{"phase": "...", "items": [Assessment...]}`。  
5. `StepEvaluation/step_n.json`：元素形如 `{"phase": "...", "value": StepEvaluation}`。  
6. `ReplanDecision/step_n.json`：元素形如 `{"phase": "...", "value": ReplanDecision}`。  

## 5.3 LLM 原始产物

1. `PlanResult/step_n_llm.json`、`StepContract/step_n_llm.json`。  
2. 每条记录结构：`stage + llm_response + error`。  
3. 用途：保留 LLM 原文，便于追溯“原始输出 -> sanitize 后结构”的差异。

## 6. 当前版本的链路结论

1. 当前 GUI-Agent 的结构主线是：  
`UIState -> PlanResult -> StepContract -> ResolvedAction -> GuardResult -> StepEvaluation -> ReplanDecision`。  
2. 结构一致性主要由 `contracts.py`、OraclePre sanitize、Policy/Expectation 共用判定模型、`to_plain_dict` 序列化共同保证。  
3. 长期记忆已停用，当前闭环依赖短期轨迹与每步契约，不依赖历史经验检索。
