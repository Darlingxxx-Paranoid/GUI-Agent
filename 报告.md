# 基于Oracle反馈驱动的GUI Agent

## 1. 整体框架修改

* 从之前的根据全局任务一次性生成子目标序列，改为ReAct框架，更加符合实际任务中的动态

* 增加CV模型，GUI环境感知更客观详细

* 加入静态兜底机制（跳变预期+CV）

## 2. 一句话总结

* **ReAct 主循环 + Oracle 证据化评估**：感知层先构建统一 UI 上下文（融合控件列表 + 可选 package/activity 等环境信号），事前 Oracle 把子目标“成功标准”结构化成可记录证据字段，事后由同一个 LLM 基于 evidence 做 before/after 语义摘要与比对，形成可复盘的闭环。

## 2. 模块解析

### 2.1 模块一：多模态环境感知层

> 负责获取并清洗当前GUI环境的客观事实，为后续所有推理和校验提供基础数据。

* **多模态状态解析**：结合底层结构树（Android `uiautomator dump` 的 XML）与轻量级CV模型（OCR 文本提取 + 组件检测 + 融合），解决单一维度提取遗漏（如纯图片广告）的问题。

* **跨模态融合（IoU 对齐）**：
  
  * 将 CV 输出的 bbox 坐标按 `resize_ratio` 还原到设备真实分辨率。
  
  * 对每个 CV 元素在 Dump 控件中做 IoU 最大匹配（阈值 0.3），融合 `resource-id/class/clickable/scrollable/content-desc/text` 等语义信息；若 Dump 文本存在则优先使用 Dump 文本。
  
  * 对未匹配但“可交互/有文本/有 content-desc”的 Dump 控件补入，弥补 CV 漏检。

* **上下文瘦身**：过滤面积过小、占屏过大（背景容器）、以及“无 text/desc/id 且不可交互且无类别”的空节点，并对过滤后控件重新编号（`[0..N-1]`），用于规划与动作映射的稳定引用。

* **LLM可消费上下文**：将融合后的控件列表序列化为紧凑文本（每控件一行：`[id]` + 类别/文本/desc/resource-id + clickable/scrollable + center），供规划与评估直接使用。

* **可复盘输出**：每次感知结束保存 `data/context/<step_x_before|after>.json`（融合后控件列表 + 元信息），便于离线对照与复盘。


### 2.2 模块二：认知规划与事前约束

> 负责“决定下一步做什么（子目标）”，并“提前设定好成功的标准和安全底线”。将规划与事前校验合并，能保证生成的子目标天生带有约束条件。

* **经验检索（长时记忆）**：在规划前，优先根据当前意图检索长期记忆库。若匹配到高相似度的成功经验，直接复用历史动作链路，跳过复杂推理。

* **ReAct 动态子目标生成**：评估“当前 UIState（控件列表摘要）”与“最终 Task 目标”的差距，不预设长链路，仅生成最近的**单个子目标**（如：点击搜索框）。子目标结构包含：
  
  * `description`：本步意图
  
  * `action_type`：tap/swipe/input/back/scroll 等
  
  * `target_widget_text/target_widget_id`：优先用控件 ID 作为定位锚点
  
  * `acceptance_criteria`：自然语言验收标准（供事前/事后语义评估）
  
  * `expected_transition`：预期跳变类型（partial_refresh/new_page/dialog/external_app）

* **事前 Oracle 生成与预期设定**：
  
  * **约束生成（结构化 PreConstraints）**：为每个子目标生成通用的“可验证约束”与“语义验收标准”，字段包括：
    
    * 页面语义类型：`source_state_type/target_state_type`（list/detail/form/dialog/search/menu/tab/selection/unknown）
    
    * 跳变预期：`transition_type/allowed_transition_types/transition_description`
    
    * 环境边界：`must_stay_in_app/expected_package/forbidden_packages/expected_activity`
    
    * 强/弱证据：`widget_should_exist/widget_should_vanish/supporting_texts/supporting_widgets/min_support_score`
    
    * 高阶语义：`semantic_criteria`（作为最终“是否完成子目标”的语义依据）
  
  * **跳变预期**：预测动作执行后App可能发生的变化类型（UI局部刷新/新Activity/拉起外部应用等），为执行瞬间提供校验基准。

* **安全拦截 (Human-in-the-loop)**：若子目标涉及“支付、删除、授权”等高风险不可逆操作，强制拦截并请求人类确认，通过后才允许流转到执行层。

### 2.3模块三：动作实例化与事中监控

> 负责将抽象的子目标转化为物理点击，并在**动作发生的瞬间产生“条件反射”式的监控**，防止系统崩溃或失控。

* **意图生成与空间对齐**：
  
  * 优先使用 `target_widget_id` 在控件列表中直接寻址；其次尝试 `target_widget_text` 进行包含匹配；若仍失败则调用 LLM 在控件列表摘要中选择目标控件 ID。
  
  * 命中控件后使用其 `center` 坐标生成可执行动作（tap/input/swipe/back 等）。

* **防死循环拦截（事中 Oracle）**：在执行前，若检测到连续3次对同一坐标/元素执行相同动作且UI无实质变化，强行熔断并打回重规划。

* **指令下发与反射监控（事中 Oracle）**：
  
  * **指令执行**：将坐标和动作类型转化为物理 ADB 指令执行。
  
  * **异常兜底**：基于执行后截图计算 HSV 像素方差，低于阈值判定可能白屏/黑屏/崩溃，建议立即重规划。
  
  * **瞬态跳变校验**：将实际的页面跳转情况与模块二中的“跳变预期”比对。若发生非预期剧变（如误点广告跳出App），立即触发拉回机制（下发全局Back指令）。

#### 2.4 模块四：事后评估与经验闭环

> 负责对动作结果进行最终的语义复盘，决定是继续前进、回溯还是终止，并沉淀经验。

* **约束检测与语义确认（以 LLM 语义比对为主）**：
  
  * **硬边界约束**：仅将“环境安全边界”作为硬失败条件，例如 `must_stay_in_app/expected_package/forbidden_packages`。
  
  * **证据收集（可能噪声）**：对 `expected_activity/require_ui_change/widget_should_exist/widget_should_vanish/supporting_*` 仅作为 evidence 汇总，不再作为一票否决条件。
  
  * **LLM 语义摘要 + 比对**：将 evidence + 执行前后 UIState 摘要一起输入同一个 LLM，让其先产出 `before_summary/after_summary/delta_summary`（页面语义、关键元素、交互态、实体信息等），再输出 `success/reason`。

* **复盘样例（建议放PPT）**：
  
  * `data/context/<step_after>.eval.json` 里包含 `constraint_evidence`（可解释证据）+ `llm_result`（before/after 摘要与判定）+ `final`（最终 success/原因），可直接用于说明“为什么判成功/失败”。

* **动态重规划与状态回溯**：
  
  * 若判定失败，LLM需分析失败原因并加入短期记忆。
  
  * 判定当前是否陷入“死胡同”（如进错详情页），若是则下发 Back 指令回溯上一层，重新观测。

* **经验沉淀**：当整个 Task 完成时，将成功的动作序列（含坐标/动作类型/描述等）持久化到长期记忆库；任务失败则不沉淀经验，避免污染。

* **事后复盘产物**：每步事后评估会额外保存 `data/context/<step_x_after>.eval.json`，包含：
  
  * 本步 `subgoal/acceptance_criteria`
  
  * 约束 evidence（含 package/activity、是否 UI 变化、强/弱证据命中等）
  
  * LLM 的语义摘要与判定结果（以及原始返回），用于离线复盘与误判定位
