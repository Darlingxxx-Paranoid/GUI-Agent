## 1. 当前框架模块（按真实执行链路）

入口：`main.py` -> `agent_loop.py:AgentLoop`

单步主链路（`AgentLoop._execute_one_step`）：

1. `Perception`：读取截图、UI dump、前台 activity/package、键盘状态，构建 `UIState`。
2. `Planner`：基于任务 + `UIState` + 短期上下文，生成 `PlanResult`。
3. `OraclePre`：将 `PlanResult` 编译为 `StepContract`（LLM 语义草案 + 规则清洗 + 失败回退）。
4. `SafetyInterceptor`：高风险关键词人类确认。
5. `ActionMapper`：将 `PlanResult + StepContract + UIState` 映射为 `ResolvedAction`；无法安全映射时抛 `MappingFailure`。
6. `OracleRuntime.pre_guard`：执行前 guard，输出 `GuardResult(phase="pre")`。
7. `ActionExecutor`：执行物理动作（支持 `launch_app`）。
8. `Perception`（再次）：得到 `new_state`。
9. `OracleRuntime.post_guard`：执行后 guard，输出 `GuardResult(phase="post")`。
10. `Evaluator`：融合 observations + policy + expectation，输出 `StepEvaluation`。
11. `Replanner`：根据 `StepEvaluation` 输出 `ReplanDecision`。
12. `Memory`：只写短期轨迹（长期记忆链路已停用）。

---

## 2. 统一契约层（Oracle/contracts.py）

`Oracle/contracts.py` 是全链路 schema 源头，负责：

1. 枚举类型定义（`ActionType/PolicyKind/FactType/Decision/...`）。
2. 核心 dataclass 定义（`StepContract/ResolvedAction/ObservationFact/StepEvaluation/...`）。
3. 解析校验（`parse_json_object`、`parse_dataclass`、`validate_action_type`）。
4. 统一序列化（`to_plain_dict`）。
5. JSON Schema 导出（`contract_schema_bundle`，当前 `schema_version="v4.0"`）。

这层保证：

1. 上游生成和下游消费使用同一字段语义。
2. LLM 输出可以被结构化校验，而不是直接透传执行。
3. 落盘、内存、日志使用一致字段展开。

3. `package_name`
4. `screen_width`

6. `keyboard_visible`
7. `screenshot_path`

## 3.3 OraclePre（契约生成）

### 3.3.1 `StepContract`（来自 `Oracle/contracts.py`，由 `Planning/oracle_pre.py` 生成）

用途：执行前策略约束 + 执行后成功判据。

字段：

1. `goal: GoalSpec`
2. `target: TargetRef | None`
3. `expectations: list[Expectation]`
4. `policies: list[PolicyRule]`
5. `planning_hints: dict`

### 3.3.2 结构字段语义

#### `GoalSpec`

1. `summary`：本步短目标。
2. `success_definition`：成功定义文本。
3. `tags`：标签。

#### `TargetRef`

1. `ref_id`：目标引用 ID。
2. `role`：`primary/secondary/fallback`。
3. `selectors`：目标匹配策略集合。
4. `resolved`：已解析目标（若有）。

#### `Selector`

1. `kind`：用于定位目标的特征类型，决定“拿什么去找控件”。  
可选值：`widget_id/resource_id/text/class_name/content_desc/bounds/point/tag`。
2. `operator`：匹配操作符，决定“怎么比对特征”。  
例如 `equals`（完全相等）、`contains`（包含）、`regex`（正则）、`overlap`（区域重叠）、`near`（距离接近）。
3. `value`：匹配值本体。  
例如文本字符串、资源 ID、坐标点 `[x,y]`、边界框 `[x1,y1,x2,y2]`。

#### `TargetResolved`

1. `widget_id`：本帧 `UIState` 中命中的控件编号（运行时 ID）。
2. `bounds`：命中控件的矩形区域 `(x1,y1,x2,y2)`。
3. `center`：命中控件中心点 `(cx,cy)`，点击类动作通常直接使用该坐标。
4. `snapshot`：命中时控件属性快照（字典，不是图片）。  
常见键包括 `text/resource_id/content_desc/class_name/clickable/enabled`，用于回放与排障。

#### `Expectation`
语义上的期望
1. `id`：期望唯一标识，用于评估结果回溯与日志定位。
2. `fact_type`：要匹配的观测事实类型（来自 `ObservationFact.type`）。
3. `scope`：期望作用范围。`anchor`=目标锚点，`local`=局部区域，`global`=全局页面。
4. `subject_ref`：期望绑定的主体引用，确保期望和目标对象一致。
5. `predicates`：字段级约束条件。  
例如限制 `attributes.text` 必须包含某词，或 `attributes.similarity` 小于阈值。
6. `weight`：匹配评分权重，用于统计指标与排序，不直接替代硬判定。
7. `optional`：是否可选。`true` 表示不命中也不构成核心失败。
8. `polarity`：期望方向。`positive`=希望出现该事实，`negative`=希望不出现该事实。
9. `tier`：重要级别。`required`=核心成功证据，`supporting`=辅助证据。

#### `PolicyRule`
过程中的合法

1. `id`：策略规则唯一标识。
2. `kind`：规则类型。  
如 `app_boundary/activity_boundary/loop_guard/visual_guard` 等。
3. `level`：违规严重度。`hard` 通常直接失败，`soft` 通常进入告警/观察，`none` 表示不升高严重度。
4. `subject_ref`：策略作用主体，和目标对象对齐。
5. `predicates`：通用扩展判定条件（字段级表达式）。
6. `tags`：策略标签，用于日志分类与诊断聚合。
7. `boundary_mode`：应用边界模式。  
`stay`=必须留在当前应用；`switch`=必须切到目标应用；`either`=两者都允许。
8. `expected_packages`：期望包名集合。  
`switch` 场景下通常要求新包名命中该集合。
9. `forbidden_packages`：禁止进入的包名集合，命中即违规。
10. `expected_activity_contains`：期望前台 Activity 名包含的子串约束。
11. `loop_threshold`：循环守卫阈值，达到后触发“重复动作”风险。
12. `max_similarity`：视觉守卫阈值，相似度过高表示界面变化不足。

说明：`extra` 字段已删除，关键策略参数全部主字段化。

## 3.4 Execution 模块

### 3.4.1 `ResolvedAction`（由 `ActionMapper` 生成）

用途：执行器可直接下发的动作。

字段：

1. `type`
2. `params`
3. `target`
4. `description`

`params` 约定：

1. `tap`：`x,y`
2. `input`：`x,y,text`
3. `swipe`：`x,y,x2,y2,duration_ms,screen_width,screen_height`
4. `long_press`：`x,y,duration_ms`
5. `launch_app`：`package, activity?`

### 3.4.2 `MappingFailure`（`Execution/action_mapper.py`）

用途：表示无法生成“安全可执行动作”。

字段：

1. `reason_code`
2. `message`

当前关键场景：`launch_app` 包名不可解析且不存在安全降级目标时抛出。

### 3.4.3 `ObservationFact`（由 `ObservationExtractor` 生成）

用途：统一事实证据载体，供 policy/evaluator 复用。

字段：

1. `fact_id`：事实唯一 ID，用于追踪这条证据在本步中的来源与引用关系。
2. `type`：事实类型，决定这条证据属于哪类变化（文本变化、包名变化、风险信号等）。
3. `scope`：作用范围。`anchor` 表示目标附近，`local` 表示局部，`global` 表示全局页面级。
4. `subject_ref`：主体引用 ID，把事实和某个目标对象绑定起来（例如 `target:primary`）。
5. `attributes`：事实细节字典，存放具体值（例如 old/new 包名、文本内容、相似度）。
6. `confidence`：事实可信度分数，当前大多由规则提取器给出固定高值。
7. `source`：事实来源模块，如 `ui_tree/screenshot_diff/runtime_guard/action_mapper`。
8. `is_derived`：是否为推导事实。`true` 通常表示由 guard 或规则推导，不是直接原始观测。
9. `evidence_refs`：外部证据引用（可选），用于把当前事实关联到其他证据 ID。

常见 `fact_type`：

1. 包/Activity 变化：`package_changed/activity_changed/package_relation_state`
2. 文本变化：`text_appeared/text_disappeared`
3. 键盘/焦点：`keyboard_changed/keyboard_state/focus_changed/focus_state`
4. 目标状态：`target_resolved/target_missing/target_presence_state/target_region_state`
5. 视觉状态：`visual_similarity_state`
6. 风险信号：`risk_detected`

### 3.4.4 `GuardResult`（由 `OracleRuntime.pre_guard/post_guard` 生成）

字段：

1. `phase: pre/post`：guard 发生在执行前还是执行后。
2. `allowed`：是否允许继续。对 `pre` 最关键，`false` 会阻断动作执行。
3. `assessments: list[Assessment]`：guard 阶段产出的策略判断结果（违规级别、原因、建议）。
4. `observations: list[ObservationFact]`：guard 阶段收集/补充的事实证据，供后续评估复用。

## 3.5 Evaluation 模块

### 3.5.1 `ExpectationMatch`（`ExpectationMatcher` 输出）

字段：

1. `expectation_id`：对应哪条 `Expectation.id`。
2. `matched`：该期望是否命中。
3. `matched_fact_ids`：命中的事实 ID 列表（来自 `ObservationFact.fact_id`）。
4. `score`：该期望的匹配得分（通常由期望权重与命中情况决定）。
5. `reason_code`：机器可解析的匹配原因码（如 matched/not_matched/negative_hit）。
6. `message`：给人看的解释文本（用于日志与诊断）。

### 3.5.2 `Assessment`

用途：统一策略/评估判定项。

字段：

1. `name`：该判定项名称（例如 `app_boundary_check`）。
2. `source`：判定来源模块，取值为 `runtime/policy_engine/evaluator/semantic_llm`。
3. `applies_to`：判定作用域，区分 `boundary/effect/semantic/safety/runtime_guard`。
4. `outcome`：判定结果，`pass/fail/uncertain`。
5. `severity`：失败严重度，`hard` 一般会触发强干预，`soft` 常进入观察或重试。
6. `reason_code`：结构化原因码，供流程分支和统计使用。
7. `message`：人类可读解释。
8. `evidence_refs`：支撑该判定的证据 ID 列表。
9. `score`：可选数值评分（并非每个判定都有）。
10. `remedy_hint: AdviceParams`：可执行建议参数，如回退步数、观察时长、原因标签。

### 3.5.3 `StepEvaluation`（Evaluator 最终输出）

字段：

1. `decision: success/fail/uncertain`：本步最终结论。
2. `confidence`：对该结论的置信度（0-1）。
3. `recommended_action`：给 replanner/主循环的建议下一步动作。
4. `assessments`：本步汇总后的判定项列表（policy + evaluator 自身判断）。
5. `observations`：本步用于评估的事实证据全集。
6. `metrics`：本步数值指标（匹配率、失败计数、相似度等）。
7. `expectation_matches`：逐条期望的匹配结果详情。

### 3.5.4 `RecommendedAction`

字段：

1. `kind`：建议动作类型，`continue/retry/replan/backtrack/observe/abort`。
2. `params: AdviceParams | None`：该建议动作的参数载荷（如回退步数、观察延时、标签）。

### 3.5.5 `AdviceParams`

字段：

1. `retry_count`：建议重试次数（通常用于 `retry`）。
2. `observe_delay_ms`：建议追加观测等待时长（毫秒）。
3. `backtrack_steps`：建议回退步数（通常用于 `backtrack`）。
4. `reason_tags`：结构化原因标签列表，便于后续策略路由与统计。
5. `target_ref`：建议动作关联的目标引用（可选）。

### 3.5.6 `NumericMetric`

字段：

1. `key`：指标名称（例如 `required_expectation_ratio`）。
2. `kind`：指标类型，`score/ratio/count/duration_ms/similarity`。
3. `value`：指标数值。
4. `unit`：单位（可选），例如 `count/facts/ms`。

## 4. 全链路中数据结构如何对齐

## 4.1 对齐原则

1. **单一契约源**：核心类型全部收敛到 `Oracle/contracts.py`。
2. **结构化输入校验**：LLM 输出先过 JSON 解析 + dataclass 校验 + action type 验证。
3. **主键语义统一**：`subject_ref` 用 `normalize_subject_ref()` 统一归一。
4. **策略字段显式化**：`PolicyRule` 关键语义主字段化（无隐式 `extra`）。
5. **统一序列化**：跨模块传递与落盘都用 `to_plain_dict()`。

## 4.2 对齐关键点

1. `Expectation.fact_type` 与 `ObservationFact.type` 一一匹配，`ExpectationMatcher` 用同一 operator 语义求值。
2. `PolicyEngine` 与 `Evaluator` 共享同一 `Assessment` 结构，保证策略结果可直接进入后评估。
3. `app_boundary` 的 `boundary_mode/expected_packages` 在 OraclePre 生成、PolicyEngine 执行、Evaluator 冲突判定三处语义一致。
4. `launch_app` 由 `ActionType` 贯穿到 `ResolvedAction(type="launch_app")`，不再“名义支持、执行降级”。

## 5. 全链路中数据结构如何生成

下表给出“谁生成、输入是什么、输出给谁”：

| 阶段 | 生成模块 | 输入 | 输出 | 消费模块 |
|---|---|---|---|---|
| 感知 | PerceptionManager | screenshot/dump/activity/package/keyboard | `UIState` | Planner / Mapper / Runtime / Evaluator |
| 规划 | Planner | task + `UIState` + short-term summary | `PlanResult` | OraclePre / Safety / Mapper |
| 预契约 | OraclePre | `PlanResult` + `UIState` + task_hint | `StepContract` | Safety / Mapper / Runtime / Evaluator |
| 映射 | ActionMapper | `PlanResult` + `StepContract` + `UIState` | `ResolvedAction` 或 `MappingFailure` | Runtime / Executor / Replanner |
| pre-guard | OracleRuntime.pre_guard | `ResolvedAction` + `StepContract` + `UIState` | `GuardResult(pre)` | AgentLoop |
| 执行 | ActionExecutor | `ResolvedAction.params` | 设备状态变化 | Perception |
| post-guard | OracleRuntime.post_guard | action + contract + old/new state | `GuardResult(post)` | Evaluator |
| 后评估 | Evaluator | contract + observations + assessments + old/new state | `StepEvaluation` | Replanner / AgentLoop |
| 重规划 | Replanner | `StepEvaluation` + `UIState` | `ReplanDecision` | AgentLoop |
| 记忆 | MemoryManager.ShortTerm | 运行轨迹 | history/action/failure | Planner 上下文 |

## 5.1 OraclePre 的生成细节

1. 先基于规则生成 fallback contract。
2. 调用 LLM 生成草案 contract。
3. 对草案做 sanitize：
   - 移除/修正非法字段
   - 保证关键 policy 存在
   - 禁止 `visual_similarity_state` 成为 required
4. 生成最终 `StepContract`。

## 5.2 ActionMapper 的生成细节

1. 常规动作：根据 target 解析结果填坐标或参数。
2. `launch_app`：按优先级解析 package/activity。
3. 解析失败时执行安全降级规则：
   - 可解析中心点 -> `tap`
   - 可唯一 selector -> `tap`
   - 都不满足 -> `MappingFailure`

## 5.3 Evaluator 的生成细节

1. 使用 `ExpectationMatcher` 得到 `ExpectationMatch`。
2. 统计 required 期望匹配情况（主判据）。
3. 融合 policy assessments。
4. 产出 `StepEvaluation` + metrics + recommendation。

---

## 6. 当前实现的语义约束（关键行为）

### 6.1 required-first 判定

`Evaluator` 主判断以 required 期望为核心，不依赖 `primary_expectation_ids`。

### 6.2 冲突分支（严格版）

`switch` 冲突仅在以下条件进入 uncertain：

1. required 未完全命中
2. `app_boundary_mode == "switch"`
3. `app_boundary_check` 为 `pass`
4. `reason_code == "switch_boundary_pass"`

不是“发生 package/activity 变化就算冲突”。

### 6.3 launch_app 禁止盲降级

当无法解析目标包且无安全 target 时，不执行默认 tap/swipe，直接失败并交由 replanner 处理。

## 8. 一句话总结

当前 GUI-Agent 已形成“`PlanResult -> StepContract -> ResolvedAction -> GuardResult -> StepEvaluation`”的统一数据闭环：结构由 `contracts.py` 约束、语义由 OraclePre/PolicyEngine/Evaluator 协同生成并对齐，且 `launch_app` 与冲突判定都落在可执行、可审计的主链路中。
