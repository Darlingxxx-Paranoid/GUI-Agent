## 1. 当前框架模块（按真实运行链路）

主入口与总控：

1. `main.py`
2. `agent_loop.py`

运行主链路（`AgentLoop._execute_one_step()`）：

1. **Perception（感知）**：截图 + dump + 当前 activity/package/键盘状态，生成 `UIState`
2. **Planning（规划）**：基于任务+UIState 输出 `PlanResult`
3. **OraclePre（前置契约生成）**：把 `PlanResult` 转为 `StepContract`
4. **SafetyInterceptor（安全拦截）**：高风险关键词人工确认
5. **ActionMapper（动作映射）**：把 `PlanResult + StepContract + UIState` 转为 `ResolvedAction`
6. **OracleRuntime.pre_guard（执行前守卫）**：输出 `GuardResult(phase=pre)`
7. **ActionExecutor（执行）**：执行 ADB 动作
8. **Perception（再次感知）**：得到 `new_state`
9. **OracleRuntime.post_guard（执行后守卫）**：输出 `GuardResult(phase=post)`
10. **Evaluator（后评估）**：输出 `StepEvaluation`
11. **Replanner（重规划决策）**：输出 `ReplanDecision`
12. **Memory（记忆）**：写短期步骤记录，任务成功后沉淀长期经验

---

## 2. 统一数据契约层（Oracle/contracts.py）

`Oracle/contracts.py` 是全链路共享 schema 源头，定义了：

1. 受控枚举：`Decision/Outcome/Severity/Scope/Phase/ActionType/PolicyKind/FactType/...`
2. 核心 dataclass：`GoalSpec/TargetRef/StepContract/ResolvedAction/GuardResult/StepEvaluation/...`
3. 解析与校验工具：`parse_json_object`、`parse_dataclass`、`validate_action_type`
4. 序列化工具：`to_plain_dict`
5. schema 导出：`contract_schema_bundle`、`export_contract_schema`

这意味着：

1. 上游模块产出的结构体字段名，和下游模块读取字段名，来自同一份定义。
2. LLM 输出要先过 schema/解析，再能进入执行链路。
3. 运行时日志与落盘 JSON 都通过 `to_plain_dict` 进行统一展开。

## 3. 各模块数据结构与字段含义

## 3.1 Perception 模块

### 3.1.1 `UIElement`（`Perception/dump_parser.py`）

来源：Android dump XML 节点。  
用途：作为融合前的语义节点。

字段：

1. `resource_id`：控件资源 ID
2. `class_name`：控件类名
3. `text`：文本
4. `content_desc`：无障碍描述
5. `bounds`：边界框 `(x1,y1,x2,y2)`
6. `clickable/scrollable/enabled/focusable/focused`：交互状态
7. `editable`：XML 原始 editable 标记
8. `checkable/checked/selected`：选择态
9. `package`：节点包名
10. `index`：层级索引

派生属性：`width/height/area/center/is_interactive/is_editable`。

### 3.1.2 `WidgetInfo`（`Perception/context_builder.py`）

来源：CV 元素与 UIElement 通过 IoU 融合后得到。  
用途：后续规划/映射/评估都用它。

字段：

1. `widget_id`：融合后重排的稳定编号（本次状态内）
2. `bounds/center`：控件空间信息
3. `category`：CV 类别
4. `text/resource_id/class_name/content_desc`：语义属性
5. `clickable/scrollable/enabled/focusable/focused/editable/checkable/checked/selected`：交互状态
6. `cv_confidence`：CV 置信度

### 3.1.3 `UIState`（`Perception/context_builder.py`）

来源：`ContextBuilder.build()`。  
用途：整个链路的“当前界面快照”。

字段：

1. `widgets: list[WidgetInfo]`
2. `activity_name`
3. `package_name`
4. `screen_width/screen_height`
5. `keyboard_visible`
6. `screenshot_path`
7. `raw_cv_elements`

辅助方法：`to_prompt_text()`（给 Planner 的 UI 文本）、`find_widget_by_text()`、`find_widget_by_id()`。

### 3.1.4 Perception 落盘结构（`data/context/*.json`）

`PerceptionManager._save_context()` 产物关键字段：

1. UI 元信息：`activity_name/package_name/screen_width/screen_height/keyboard_visible`
2. 来源路径：`screenshot_path/dump_path`
3. 统计：`widgets_count/editable_widgets_count/focused_widgets_count`
4. 详细控件：`widgets[]`（来自 `WidgetInfo.to_dict()` + `cv_confidence`）

## 3.2 Planning 模块

### 3.2.1 `PlanResult`（`Planning/planner.py`）

来源：经验回放或 LLM 规划。  
用途：下一步意图的标准表示。

字段：

1. `goal: GoalSpec`
2. `target: TargetRef | None`
3. `requested_action_type: str`
4. `input_text: str`
5. `planning_hints: dict`
6. `is_task_complete: bool`
7. `reasoning: str`
8. `from_experience: bool`

### 3.2.2 Planner 的 LLM 输出 schema（严格）

Planner 要求顶层仅允许以下 key：

1. `is_task_complete`
2. `reasoning`
3. `goal`
4. `requested_action_type`
5. `input_text`
6. `target`
7. `planning_hints`

校验机制：

1. `parse_json_object()`：清洗并解析 JSON
2. 顶层字段白名单检查
3. `goal/target` 用 `parse_dataclass(..., strict=True)`
4. `requested_action_type` 用 `validate_action_type()`

说明：`ActionType` 虽有 `launch_app`，但 Planner 当前把它降级为 `tap`，避免进入不支持分支。

## 3.3 OraclePre 模块

### 3.3.1 `StepContract`

来源：`OraclePre.generate_contract(plan, ui_state, task_hint)`。  
用途：执行前约束与评估基准。

字段：

1. `goal: GoalSpec`
2. `target: TargetRef | None`
3. `expectations: list[Expectation]`
4. `policies: list[PolicyRule]`
5. `planning_hints: dict`

### 3.3.2 子结构字段语义

`GoalSpec`：

1. `summary`：本步目标摘要
2. `success_definition`：本步成功定义
3. `tags`：标签（不用于硬分支）

`TargetRef`：

1. `ref_id`：主体引用 ID（如 `target:primary` / `widget:12`）
2. `role`：`primary/secondary/fallback`
3. `selectors`：候选选择器列表
4. `resolved`：当前已解析到的控件位置信息

`Selector`：

1. `kind`：`widget_id/text/resource_id/content_desc/class_name/bounds/point/tag`
2. `operator`：`equals/contains/overlap/...`
3. `value`：匹配值

`TargetResolved`：

1. `widget_id`
2. `bounds`
3. `center`
4. `snapshot`（目标控件快照）

`Expectation`：

1. `id`
2. `fact_type`
3. `scope`
4. `subject_ref`
5. `predicates: list[Predicate]`
6. `weight`
7. `optional`
8. `polarity`（`positive/negative`）
9. `tier`（`required/supporting`）

`Predicate`：

1. `field`（如 `attributes.similarity`）
2. `operator`
3. `value`

`PolicyRule`：

1. `id`
2. `kind`（如 `loop_guard/app_boundary/activity_boundary/visual_guard`）
3. `level`（`none/soft/hard`）
4. `subject_ref`
5. `predicates`
6. `tags`
7. `extra`（边界规则参数，如 `expected_package`）

## 3.4 Execution 模块

### 3.4.1 `ResolvedAction`（`Execution/action_mapper.py`）

来源：`ActionMapper.map_action(...)`。  
用途：可直接执行的动作。

字段：

1. `type`：动作类型
2. `params`：执行参数
3. `target`：已解析目标
4. `description`：动作说明

`params` 约定（当前实现）：

1. `tap`：`x,y`
2. `input`：`x,y,text`
3. `swipe`：`x,y,x2,y2,duration_ms,screen_width,screen_height`
4. `back/enter`：空字典
5. `long_press`：`x,y,duration_ms`

说明：`launch_app` 在映射阶段也会降级为 `tap`。

### 3.4.2 `GuardResult`（`Execution/oracle_runtime.py`）

来源：`pre_guard/post_guard`。  
用途：执行前后守卫结果。

字段：

1. `phase`：`pre/post`
2. `allowed`：pre 阶段是否允许继续执行
3. `assessments: list[Assessment]`
4. `observations: list[ObservationFact]`

### 3.4.3 `ObservationFact`（`Execution/observation_extractor.py`）

来源：old/new UIState 差异 + action 上下文。  
用途：供 PolicyEngine 和 Evaluator 统一消费。

字段：

1. `fact_id`
2. `type`（如 `package_changed/text_appeared/visual_similarity_state/...`）
3. `scope`
4. `subject_ref`
5. `attributes`
6. `confidence`
7. `source`
8. `is_derived`
9. `evidence_refs`

典型事实：

1. 包/Activity/键盘/焦点变化
2. 文本出现/消失
3. 视觉相似度状态
4. 目标是否仍可解析（`target_resolved/target_missing/target_presence_state`）

### 3.4.4 `Assessment`（运行守卫与策略引擎）

字段：

1. `name`
2. `source`（`runtime/policy_engine/evaluator/semantic_llm`）
3. `applies_to`（`boundary/effect/semantic/safety/runtime_guard`）
4. `outcome`（`pass/fail/uncertain`）
5. `severity`（`none/soft/hard`）
6. `reason_code`
7. `message`
8. `evidence_refs`
9. `score`
10. `remedy_hint: AdviceParams | None`

`AdviceParams` 字段：

1. `retry_count`
2. `observe_delay_ms`
3. `backtrack_steps`
4. `reason_tags`
5. `target_ref`

## 4. 全链路中数据结构如何对齐与生成

## 4.1 对齐总原则

1. **统一 schema 源头**：`Oracle/contracts.py`
2. **入口严格校验**：Planner 的 LLM 输出必须通过 `parse_json_object + parse_dataclass + validate_action_type`
3. **运行中统一对象**：后续模块直接传递 dataclass（`PlanResult -> StepContract -> ResolvedAction -> GuardResult -> StepEvaluation`）
4. **统一序列化**：写内存和落盘都经 `to_plain_dict`

## 4.2 链路分步（谁生成，谁消费）

| 阶段 | 生成方 | 主要输入 | 主要输出 | 下游消费方 |
|---|---|---|---|---|
| 感知 | `PerceptionManager` | screenshot + dump + activity/package/keyboard | `UIState` | Planner / ActionMapper / OracleRuntime / Evaluator |
| 规划 | `Planner` | `task + UIState + short_term + long_term` | `PlanResult` | OraclePre / Safety / ActionMapper |
| 预契约 | `OraclePre` | `PlanResult + UIState` | `StepContract` | Safety / ActionMapper / OracleRuntime / Evaluator |
| 动作映射 | `ActionMapper` | `PlanResult + UIState + StepContract` | `ResolvedAction` | OracleRuntime / AgentLoop执行器 |
| 执行前守卫 | `OracleRuntime.pre_guard` | `ResolvedAction + StepContract + UIState` | `GuardResult(pre)` | AgentLoop（阻断/继续） |
| 执行后守卫 | `OracleRuntime.post_guard` | `ResolvedAction + StepContract + old/new UIState` | `GuardResult(post)` | Evaluator |
| 后评估 | `Evaluator` | `StepContract + ResolvedAction + old/new UIState + GuardResult(post)` | `StepEvaluation` | Replanner / AgentLoop |
| 重规划决策 | `Replanner` | `StepEvaluation + UIState` | `ReplanDecision` | AgentLoop（back/retry/replan/abort） |
| 经验沉淀 | `Replanner.save_task_experience` | `ShortTermMemory.history` | `ExperienceRecord(step_triplets)` | 下一次 Planner 经验回放 |

## 4.3 关键对齐机制细节

1. `subject_ref` 对齐：通过 `normalize_subject_ref(...)` 统一目标主体 ID，避免 expectation/policy/fact 的主体漂移。
2. `fact_type` 对齐：ObservationExtractor 生成受控 `fact_type`，ExpectationMatcher 按同名匹配。
3. `predicate` 对齐：ExpectationMatcher 与 PolicyEngine 都使用统一 operator 语义（`equals/contains/gt/...`）。
4. `action_type` 对齐：先 `validate_action_type`；当前 `launch_app` 统一降级成 `tap`，避免执行器无实现时结构失配。
5. `pre/post` 边界对齐：pre_guard 负责阻断，post_guard 负责提供事实与评估素材，最终 `success/fail/uncertain` 只由 `StepEvaluation.decision` 给出。
6. 落盘对齐：context、eval、memory 都保存“同名字段展开字典”，便于回放/审计/回归比较。

## 5. 当前实现中的“设计-运行”差异（与数据结构相关）

1. `prompt/action_mapper_prompt.py`、`prompt/oracle_pre_prompt.py`、`prompt/evaluator_prompt.py`、`prompt/replanner_prompt.py` 已存在，但当前主链路中实际只有 **Planner** 在调用 LLM。
2. `ActionType` 合同含 `launch_app`，但 Planner 与 ActionMapper 都主动降级为 `tap`，主循环执行分支也未专门处理 `launch_app`。
3. 因此当前“数据结构对齐”是通过受控降级保持一致，而不是端到端真实执行 `launch_app`。

---

## 6. 一句话总结

当前 GUI-Agent 的数据链路已经以 `Oracle/contracts.py` 为统一 contract 中心，形成了 `UIState -> PlanResult -> StepContract -> ResolvedAction -> GuardResult -> StepEvaluation -> ReplanDecision/ExperienceRecord` 的闭环；字段对齐主要靠 **受控枚举 + dataclass 严格解析 + 统一序列化** 三层机制保证。
