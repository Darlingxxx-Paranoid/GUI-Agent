# 基于Oracle反馈驱动的GUI Agent

## 1. 整体框架修改

* 从之前的根据全局任务一次性生成子目标序列，改为ReAct框架，更加符合实际任务中的动态

* 增加CV模型，GUI环境感知更客观详细

* 加入静态兜底机制（跳变预期+CV）

## 2. 模块解析

### 2.1 模块一：多模态环境感知层

> 负责获取并清洗当前GUI环境的客观事实，为后续所有推理和校验提供基础数据。

* **多模态状态解析**：结合底层结构树（Dump JSON）与轻量级CV模型（OCR文本提取、标准Icon检测），解决单一维度提取遗漏（如纯图片广告）的问题。

* **上下文瘦身**：进行数据清洗，过滤不可见、不可交互的冗余元素，生成精简的实时 JSON 数据，有效压缩 LLM 的上下文窗口占用。

### 2.2 模块二：认知规划与事前约束

> 负责“决定下一步做什么（子目标）”，并“提前设定好成功的标准和安全底线”。将规划与事前校验合并，能保证生成的子目标天生带有约束条件。

* **经验检索（长时记忆）**：在规划前，优先根据当前意图检索长期记忆库。若匹配到高相似度的成功经验，直接复用历史动作链路，跳过复杂推理。

* **ReAct 动态子目标生成**：评估“当前UI状态”与“最终Task目标”的差距，不预设长链路，仅生成最近的**单个子目标**（如：点击搜索框），以应对突发路由变化。

* **事前 Oracle 生成与预期设定**：
  
  * **约束生成**：强制LLM为该子目标生成“验收标准”，与目标控件属性（如特定文字内容）或视觉特征（如“出现支付成功”）强绑定。
  
  * **跳变预期**：预测动作执行后App可能发生的变化类型（UI局部刷新/新Activity/拉起外部应用等），为执行瞬间提供校验基准。

* **安全拦截 (Human-in-the-loop)**：若子目标涉及“支付、删除、授权”等高风险不可逆操作，强制拦截并请求人类确认，通过后才允许流转到执行层。

### 2.3模块三：动作实例化与事中监控

> 负责将抽象的子目标转化为物理点击，并在**动作发生的瞬间产生“条件反射”式的监控**，防止系统崩溃或失控。

* **意图生成与空间对齐**：基于子目标，输出动作类型及目标控件特征，在感知层的控件列表中寻址获取精确坐标。

* **防死循环拦截（事中 Oracle）**：在执行前，若检测到连续3次对同一坐标/元素执行相同动作且UI无实质变化，强行熔断并打回重规划。

* **指令下发与反射监控（事中 Oracle）**：
  
  * **指令执行**：将坐标和动作类型转化为物理 ADB 指令执行。
  
  * **异常兜底**：利用CV实时检测是否出现App崩溃、白屏或黑屏，如有直接触发报警。
  
  * **瞬态跳变校验**：将实际的页面跳转情况与模块二中的“跳变预期”比对。若发生非预期剧变（如误点广告跳出App），立即触发拉回机制（下发全局Back指令）。

#### 2.4 模块四：事后评估与经验闭环

> 负责对动作结果进行最终的语义复盘，决定是继续前进、回溯还是终止，并沉淀经验。

* **约束检测与语义确认**：比对动作后的新UI状态与模块二设定的“事前约束”。只有基础约束通过，才调用LLM进行子目标的高阶语义一致性确认（即判断目标是否真正完成）。

* **动态重规划与状态回溯**：
  
  * 若判定失败，LLM需分析失败原因并加入短期记忆。
  
  * 判定当前是否陷入“死胡同”（如进错详情页），若是则下发 Back 指令回溯上一层，重新观测。

* **经验沉淀**：对于成功完成的子目标，从任务队列中移除。当整个Task完成时，将正确的动作序列转化为长期记忆，持久化存储到长期记忆库里
